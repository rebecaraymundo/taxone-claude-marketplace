# Cargill - Duplicidade visual de registros no log OBI

## Root Cause

O package `LIB_IMPORT.pck` possui dois blocos MERGE INTO DET_PROC_IMP que atualizam contadores
após a importação. Os MERGEs somavam `qtde_regs_dupl` (quantidade de registros duplicados) tanto
em `tot_ignorados` quanto em `tot_lidos`:

```sql
MERGE INTO DET_PROC_IMP a
USING (SELECT num_processo, NVL(qtde_regs_dupl, 0) AS qtde_regs_dupl
         FROM DET_JOB_IMPORT ...) b
   ON (a.num_processo = b.num_processo)
WHEN MATCHED THEN UPDATE SET
  a.tot_lidos = a.tot_lidos + b.qtde_regs_dupl,        -- BUG: inflava tot_lidos
  a.tot_ignorados = a.tot_ignorados + b.qtde_regs_dupl; -- OK
```

O campo `tot_lidos` já continha a contagem real de registros lidos pela procedure de importação.
Somar `qtde_regs_dupl` inflava artificialmente o total, causando divergência visual no log OBI.

## Solucao

Remover a linha `a.tot_lidos = a.tot_lidos + b.qtde_regs_dupl` dos 2 blocos MERGE
(linhas ~1634 e ~1927). Manter apenas a soma em `tot_ignorados`.

Fix: **2 linhas removidas**, zero linhas adicionadas.

## Analise DBA

- **Trigger TRG_DET_PROC_IMP_T1**: AUTONOMOUS_TRANSACTION, não dispara no fluxo LIB_IMPORT
  (num_lote já setado via LIB_GLOBAL.set_num_lote antes do MERGE)
- **Índice recomendado**: `DET_JOB_IMPORT(NUM_JOB, NUM_PROCESSO)` — verificar existência no banco
- **Constraints**: Nenhuma CHECK/FK depende de tot_lidos
- **Dados históricos**: Script de correção fornecido (subtrair qtde_regs_dupl do tot_lidos/QTD_LIDO
  em DET_PROC_IMP e PROTOCOLO_LOTE_IMP_TOT)

## Licoes Aprendidas

1. **MERGE com múltiplos SET**: Quando um MERGE atualiza vários campos, verificar se CADA campo
   realmente precisa receber a mesma soma. Neste caso, duplicados devem incrementar `tot_ignorados`
   mas NÃO `tot_lidos` (já contabilizados).

2. **Duplicação de blocos**: O mesmo MERGE existia em 2 locais (fluxo principal e alternativo).
   Bugs em código duplicado tendem a ser replicados. Ao corrigir, sempre buscar TODOS os blocos
   similares (grep pelo pattern SQL).

3. **Padrão LIB_IMPORT**: A estrutura é `importação → contagem → MERGE → log`. O MERGE serve
   para AJUSTAR contadores, não para RECALCULAR. Somas no MERGE devem ser aditivas apenas para
   campos que ainda não contêm o valor.

4. **Trigger AUTONOMOUS_TRANSACTION**: Não causa deadlock com o MERGE porque opera em sessão
   separada e em tabela diferente (PROTOCOLO_LOTE_IMP_TOT). Porém, pode causar lock wait em
   cenários de importação avulsa (TAX-AUTOMATION com num_lote NULL).
