# DIME SC - Erro de conversao COD_CLASSE nao-numerico

## Root Cause

A colecao `RG_CLASSE_TP_QUAD_ITEM` no package `EST_DIME_SC_FPROC` e declarada como
`TYPE TVETOR IS TABLE OF NUMBER(17, 2) INDEX BY BINARY_INTEGER` (linha 10).

O cursor `GIA_INFO` (linhas 2157-2163) busca `COD_CLASSE` (VARCHAR2) da tabela `EST_GIA_INFO`
e o usa diretamente como indice da colecao no loop (linha 5667):
```sql
IF EST_DIME_SC_FPROC.RG_CLASSE_TP_QUAD_ITEM.EXISTS(C.COD_CLASSE) THEN
```

O Oracle faz conversao implicita de VARCHAR2 para BINARY_INTEGER. Quando um registro em
`EST_GIA_INFO` contem `COD_CLASSE` com valor nao-numerico (inserido manualmente via TFix),
essa conversao falha com ORA-06502: "numeric or value error: character to number conversion error".

**Causa da insercao de dados sujos:** A tabela `EST_GIA_INFO` e alimentada pela procedure
`SAF_EST_GIA_INFO` (arquivo `s_oagia.sql`) que grava COD_CLASSE como strings numericas validas.
Porem, a tabela tambem pode ser alimentada por TFix (insercao manual), sem validacao de tipo.

## Solucao

Dupla camada de protecao no package `EST_DIME_SC_FPROC`:

### 1. Filtro no cursor GIA_INFO (linha 2163)
Adicionado `AND REGEXP_LIKE(COD_CLASSE, '^\d+$')` no WHERE do cursor para excluir registros
com COD_CLASSE nao-numerico diretamente na camada SQL:

```sql
CURSOR GIA_INFO IS
  SELECT COD_CLASSE, SUM(VLR_OUTROS) VALOR
    FROM EST_GIA_INFO
   WHERE COD_EMPRESA = P_COD_EMPRESA
     AND COD_ESTAB = P_COD_ESTAB
     AND DAT_APURACAO BETWEEN P_DATA_INI AND P_DATA_FIM
     AND REGEXP_LIKE(COD_CLASSE, '^\d+$')
   GROUP BY COD_CLASSE;
```

### 2. Validacao defensiva no loop (linhas 5667-5674)
Substituido acesso direto por validacao REGEXP_LIKE + TO_NUMBER explicito:

```sql
FOR C IN GIA_INFO LOOP
  IF C.COD_CLASSE IS NOT NULL AND REGEXP_LIKE(C.COD_CLASSE, '^\d+$') THEN
    IF EST_DIME_SC_FPROC.RG_CLASSE_TP_QUAD_ITEM.EXISTS(TO_NUMBER(C.COD_CLASSE)) THEN
      SetV(EST_DIME_SC_FPROC.RG_CLASSE_TP_QUAD_ITEM(TO_NUMBER(C.COD_CLASSE)),
           C.VALOR);
    END IF;
  END IF;
END LOOP;
```

## Pattern: Protecao contra conversao implicita VARCHAR2 → BINARY_INTEGER

**Quando aplicar:** Sempre que uma coluna VARCHAR2 for usada como indice de uma colecao
`INDEX BY BINARY_INTEGER` ou `INDEX BY PLS_INTEGER`.

**Como detectar:** Buscar no codigo por `.EXISTS(campo_varchar)` onde a colecao e indexada por inteiro.

**Solucao padrao:**
```sql
-- Camada SQL (cursor): filtrar na origem
AND REGEXP_LIKE(campo, '^\d+$')

-- Camada PL/SQL (loop): validar antes de usar
IF campo IS NOT NULL AND REGEXP_LIKE(campo, '^\d+$') THEN
  colecao.EXISTS(TO_NUMBER(campo))
END IF;
```

**Performance:** REGEXP_LIKE no WHERE do cursor e negligivel quando ja existem filtros de igualdade
(COD_EMPRESA, COD_ESTAB) que reduzem o volume significativamente.

## Licoes Aprendidas

1. **Colecoes INDEX BY BINARY_INTEGER com chave VARCHAR2 sao armadilhas silenciosas.**
   O Oracle faz conversao implicita que funciona ate encontrar dados sujos. Sempre usar
   TO_NUMBER explicito com protecao REGEXP_LIKE.

2. **TFix e uma fonte de dados sujos.** Tabelas alimentadas por TFix (insercao manual)
   podem conter valores fora do dominio esperado. Codigo PL/SQL que consome essas tabelas
   deve ser defensivo.

3. **Dupla camada de protecao e o padrao correto.** Filtrar no cursor (eficiencia) +
   validar no loop (seguranca). Se o cursor for alterado no futuro, a validacao no loop
   ainda protege.

4. **A descricao do WI mencionava "linha 5507" que na verdade corresponde a ~5667.**
   Numeros de linha reportados pelo cliente/suporte podem estar desatualizados em relacao
   ao repositorio atual. Sempre buscar pelo contexto (nome de variavel, padrao de codigo).

5. **EST_GIA_INFO e usada por GIAs de 37 arquivos (multiplos estados).** Verificar se
   outros estados tem o mesmo padrao vulneravel de acesso a COD_CLASSE com colecao
   INDEX BY BINARY_INTEGER. Potencial para correcao preventiva em lote.
