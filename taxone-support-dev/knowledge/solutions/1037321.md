# Universal ID extenso nao carrega lista de empresas

## Root Cause

O campo `cod_usuario` era definido como `VARCHAR2(40)` em multiplos pontos do sistema:
- Variaveis globais (LIB_BD.pck, LIB_GLOBAL.pck)
- Tipos Oracle (USUARIO_EMP_OT.typ, USUARIO_ESTAB_OT.typ)
- Tabelas de mapeamento (usuario_empresa_man, usuario_estab_man)

Quando um usuario com Universal ID de 46+ caracteres (ex: `luisa.ferreirafournier@external.stellantis.com`)
era cadastrado, o ID era **truncado silenciosamente** para 40 chars nas variaveis/tipos Oracle.
O cadastro do usuario funcionava (tabela principal aceita 100 chars), mas ao buscar
empresas/estabelecimentos vinculados, o match falhava porque o ID truncado nao correspondia
ao ID completo na tabela de usuarios.

## Solucao

Expandido `cod_usuario` de `VARCHAR2(40)` para `VARCHAR2(100)` em:
- `LIB_BD.pck` — variavel global
- `LIB_GLOBAL.pck` — variavel global
- `USUARIO_EMP_OT.typ` — tipo Oracle
- `USUARIO_ESTAB_OT.typ` — tipo Oracle

Script DDL (`DDL_MFS1037321_T1.sql`) para alterar colunas nas tabelas:
- `usuario_empresa_man.cod_usuario` → VARCHAR2(100)
- `usuario_estab_man.cod_usuario` → VARCHAR2(100)

## Licoes Aprendidas

1. **Truncamento silencioso em VARCHAR2**: Oracle trunca silenciosamente valores quando
   atribuidos a variaveis com tamanho menor. Nao gera erro, apenas perde dados.
   Sempre usar `%TYPE` para herdar tamanho da coluna, ou garantir consistencia manual.

2. **Consistencia de tamanhos**: Quando um campo aparece em tabela, tipo Oracle, variavel
   global e parametro de funcao, TODOS devem ter o mesmo tamanho. Buscar por
   `cod_usuario.*VARCHAR2` para encontrar todas as ocorrencias.

3. **DDL necessario**: Correcoes que envolvem tamanho de coluna requerem script DDL para
   as tabelas alem das mudancas em PL/SQL. Sempre incluir o DDL no PR.

4. **Universal ID moderno**: Com SSO federado (Stellantis, etc.), emails corporativos
   podem ter 60-80+ caracteres. VARCHAR2(100) e um tamanho seguro.
