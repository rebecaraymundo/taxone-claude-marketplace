# SOUZA CRUZ - Performance LIB_IMPORT cursor c_prog UNION ALL

## Root Cause

O package LIB_IMPORT.pck (Job Servidor / Importacao Automatica) possui cursor
c_prog (linhas 680-772) que carrega lista de tabelas a processar. O cursor usa
UNION entre duas subqueries filtradas por `ind_multi_empresa`:

- Query 1: `ind_multi_empresa IS NULL OR ind_multi_empresa = 'N'`
- Query 2: `ind_multi_empresa = 'S'`

Condicoes mutuamente exclusivas — UNION forca SORT UNIQUE desnecessario sobre
resultado potencialmente grande, consumindo TEMP tablespace e CPU.

## Solucao

Converter UNION para UNION ALL (linha 731). Fix de 1 palavra.

Pattern identico ao WI #1052913 (DMED, 15 conversoes) e WI #1028814 (EFD PIS/COFINS,
6 conversoes).

## Analise Complementar

O WI reporta problema de "arquivos nao consumidos no FileZilla" que e primariamente
de infraestrutura/conectividade. A otimizacao reduz tempo de processamento geral
do Job Servidor, mas pode nao resolver completamente o problema reportado.

Outras oportunidades de otimizacao identificadas (NAO implementadas por risco):
- UPDATE dinamico com 3 subqueries correlacionadas IN (linhas 1284-1296)
- Nested loop sobre RELAC_TAB_GRUPO (linhas 875-887)
- Indices recomendados em DET_PROC_IMP, LOG_DET_PROC

## Licoes Aprendidas

1. **LIB_IMPORT como hotspot de performance**: Package central do Job Servidor,
   processado para CADA empresa/estabelecimento. Otimizacoes aqui tem impacto
   multiplicativo.

2. **Escopo conservador**: Para WIs de performance em packages criticos, aplicar
   apenas fixes de baixo risco (UNION ALL) e documentar otimizacoes adicionais
   para avaliacao futura.

3. **ind_multi_empresa pattern**: Flag booleano NULL/'N'/'S' com condicoes
   exclusivas. Mesmo pattern pode existir em outros cursores similares.
