# AEGEA - ECD nao gera registros I200/I250

## Root Cause

No package `SAF_SPED_CONTABIL_FPROC.pck`, o procedimento `reg_i200_i250` possui dois loops
aninhados para gerar registros I200 (lancamentos) e I250 (partidas). Um fix anterior (MFS534383)
adicionou validacao de conta inativa dentro do loop INTERNO (I250).

O problema: quando uma partida tinha conta inativa sem versao ativa correspondente, o codigo
executava `RAISE Prox_Lancto_w` dentro de um bloco `BEGIN/EXCEPTION/END`. A excecao propagava
para o handler do loop EXTERNO, que fazia FETCH do proximo lancamento. Resultado:

- **TODAS** as partidas restantes do lancamento atual eram descartadas
- O lancamento inteiro (I200 + I250s) era perdido
- Para clientes com contas inativas, lancamentos validos nao apareciam no ECD

```
Loop EXTERNO (I200) -> FETCH lancamento
  Loop INTERNO (I250) -> processa partidas
    BEGIN
      SELECT count(1) INTO Exist_w ... WHERE Ind_Situacao = 'A';
      IF Exist_w = 0 THEN
        RAISE Prox_Lancto_w;  <- ESCAPAVA AMBOS OS LOOPS (bug)
      END IF;
    EXCEPTION
      WHEN NO_DATA_FOUND THEN
        RAISE Prox_Lancto_w;  <- IDEM
    END;
```

## Solucao

Reestruturar o codigo para usar pattern **flag + GOTO** ao inves de RAISE:

```sql
-- DEPOIS (corrigido):
BEGIN
  SELECT count(1) INTO Exist_w ... WHERE Ind_Situacao = 'A';
EXCEPTION
  WHEN NO_DATA_FOUND THEN
    Exist_w := 0;
END;
IF Exist_w = 0 THEN
  GOTO skip_partida_mfs1058225;   -- Pula apenas a partida, nao o lancamento
END IF;
... (processamento da partida I250) ...
<<skip_partida_mfs1058225>>
NULL;   -- Obrigatorio: label nao pode ser ultima instrucao antes de END LOOP
```

Mesma correcao aplicada no loop externo (I200) e interno (I250).

**Importante**: GOTO nao pode estar DENTRO de um bloco EXCEPTION em PL/SQL (PLS-00103).
A solucao separa o SELECT em um bloco proprio e coloca o GOTO no nivel do loop.

Fix: **15 insercoes, 23 delecoes**.

## Licoes Aprendidas

1. **RAISE dentro de loops aninhados**: Em PL/SQL, RAISE propaga ate o handler mais proximo
   que captura a excecao. Se o handler esta no loop EXTERNO, o RAISE escapa AMBOS os loops.
   Sempre verificar em qual nivel o handler esta quando usar RAISE em loops aninhados.

2. **GOTO vs RAISE para skip**: Para pular uma iteracao (CONTINUE-like), usar:
   - `GOTO label` + `NULL;` antes de `END LOOP` (PL/SQL nao tem CONTINUE em versoes antigas)
   - Nunca RAISE para skip, pois propagacao e imprevisivel com BEGIN/EXCEPTION/END aninhados

3. **GOTO nao pode sair de EXCEPTION**: Oracle PL/SQL nao permite GOTO de dentro de um
   EXCEPTION handler para fora do bloco. Solucao: separar SELECT em bloco proprio, usar flag
   (Exist_w), e GOTO no nivel do loop.

4. **NULL; apos label**: Em PL/SQL, um label nao pode ser a ultima instrucao antes de END LOOP.
   Sempre adicionar `NULL;` apos o label como placeholder.

5. **Fixes sobre fixes**: MFS534383 introduziu a validacao de conta inativa (correto), mas usou
   RAISE que tinha efeito colateral (incorreto). Ao revisar fixes anteriores, verificar se o
   mecanismo de controle de fluxo esta correto, nao apenas a logica de negocio.
