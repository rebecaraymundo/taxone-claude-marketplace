# ERICSSON/MODEC/WABTEC - ECD nao gera registros I350/I355 em determinados dias

## Root Cause

Duas causas complementares no package `SAF_SPED_CONTABIL_FPROC.pck`:

### Causa 1 (principal): RAISE descartava lancamento inteiro de encerramento

No procedimento `reg_i200_i250`, quando uma partida (I250) de um lancamento de encerramento
(TIPO_LANCTO = 'E') possuia conta com `IND_SITUACAO <> 'A'` (inativa), o codigo executava
`RAISE Prox_Lancto_w`. A excecao propagava para o handler do loop EXTERNO, descartando o
lancamento INTEIRO — incluindo partidas validas.

Consequencia: `grava_tmp_sped_i310_i355` nunca era chamada para esse lancamento, entao
`vlr_encerramento` permanecia NULL na `SPED_I310_I355_WORK`. Na procedure
`reg_i300_i310_i350_i355`, `SUM(vlr_encerramento)` retornava NULL, `ind_encer_w` permanecia
'N', e I350/I355 nao era gerado para aquela data.

O comportamento intermitente explica-se porque:
- Em execucoes diferentes, o Oracle pode processar partidas em ordens diferentes
- Se a partida com conta inativa e processada ANTES das partidas de encerramento validas,
  o RAISE descarta tudo
- Se processada DEPOIS, as partidas de encerramento ja foram gravadas na work table

### Causa 2 (complementar): Cursor c_Encerr sem ORDER BY

O cursor `c_Encerr` (linhas 8187-8206) le da `SPED_I310_I355_WORK` com GROUP BY mas SEM
ORDER BY. Este cursor alimenta o pre-carregamento de dados de encerramento para BP/DRE.
Sem ORDER BY, a ordem de processamento era nao-deterministica, podendo gerar resultados
inconsistentes entre execucoes.

**Nota:** O cursor principal `c_i300_350` (que gera I350/I355 efetivamente) JA possuia
ORDER BY correto. A causa raiz original descrita no PM ("ORDER BY ausente") se aplicava
apenas ao cursor `c_Encerr`, nao ao cursor principal.

## Solucao

### Correcao 1: RAISE → GOTO skip_partida (cherry-pick MFS1058225)

```sql
-- ANTES:
IF Exist_w = 0 THEN
  RAISE Prox_Lancto_w;  -- Descartava lancamento inteiro
END IF;

-- DEPOIS:
IF Exist_w = 0 THEN
  FETCH c_lancto INTO r_lancto;
  GOTO skip_partida_mfs1058225;  -- Pula apenas a partida inativa
END IF;
... (processamento normal) ...
<<skip_partida_mfs1058225>>
NULL;
```

### Correcao 2: ORDER BY no cursor c_Encerr

```sql
-- ANTES:
GROUP BY Data_Lancto, Cod_Conta, Grupo_Conta, Cod_Custo, Grupo_Custo
) LOOP

-- DEPOIS:
GROUP BY Data_Lancto, Cod_Conta, Grupo_Conta, Cod_Custo, Grupo_Custo
ORDER BY Data_Lancto, Cod_Conta, Grupo_Conta, Cod_Custo, Grupo_Custo
) LOOP
```

## Licoes Aprendidas

1. **Falha silenciosa e a mais perigosa**: O Oracle gerou o arquivo ECD com "sucesso" mas
   com dados incompletos. Nenhum log de erro, nenhuma excecao — apenas registros faltantes.
   Considerar adicionar log de WARNING quando I350/I355 nao e gerado para data com
   lancamento de encerramento esperado.

2. **Causa raiz vs sintoma**: A causa raiz descrita no PM ("ORDER BY ausente") era
   parcialmente correta (c_Encerr), mas a causa PRINCIPAL era outra (RAISE no loop).
   Sempre validar a causa raiz com analise de codigo, nao assumir.

3. **Multiplos clientes simultaneos = bug real**: Quando 3+ clientes reportam o mesmo
   problema em periodo proximo, descartar "parametrizacao do cliente" como causa, mesmo
   que tickets historicos similares tenham sido resolvidos assim. O padrao historico pode
   mascarar um bug real.

4. **Cherry-pick como estrategia**: Quando uma correcao existente (MFS1058225) resolve
   parte do problema, incorporar via cherry-pick ao inves de reimplementar. Evita
   conflitos e mantém rastreabilidade.

5. **ORDER BY defensivo em work tables**: Sempre adicionar ORDER BY explicito em cursores
   que leem de work tables temporarias, mesmo que o resultado do GROUP BY pareca suficiente.
   O Oracle nao garante ordem sem ORDER BY, independente de indices.
