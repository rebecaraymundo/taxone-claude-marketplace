# Informe Rendimentos 3208+1889 gera informes separados para mesmo beneficiario

## Root Cause

O sistema gera informes separados para codigos 3208 (Alugueis/royalties PF) e 1889
(RRA - Rendimentos Recebidos Acumuladamente) do mesmo beneficiario (CPF). O correto
e gerar UM informe consolidado contendo ambos os blocos (aba 1 para 3208, aba 6 para 1889).

O problema ocorre em 4 camadas:

1. **PL/SQL** (`pkg_comprovante_retencoes.pck`): O cursor `cRegsListaCompAnual` exporta
   `grupo_darf as grupo_retencao` no CSV. Codigos 3208 e 1889 tem `grupo_darf` diferentes.

2. **CSV Export**: Gera linhas separadas por (CPF + grupo_retencao), resultando em 2 linhas
   para o mesmo beneficiario quando possui ambos os codigos.

3. **Java** (`CedulaDDAO.obterInformes()` linhas 134-203): Agrupa registros do CSV por CPF.
   Quando o mesmo CPF aparece em linhas separadas (grupos diferentes), cria objetos `CedulaD`
   distintos, gerando informes separados.

4. **Jasper** (`rel_irpf_cpf.jrxml` linhas 95-98): Filtra por
   `irt_cedula_d.grupo_retencao = $P{pGrupoRetencao}`, mostrando apenas dados do grupo
   especifico de cada informe.

## Contexto: Fix Anterior WI #1061569

O fix do WI #1061569 adicionou `grupoRetencao` na comparacao de duplicidade para SEPARAR
corretamente 3208 e 5204. O #1046488 e o caso inverso: 3208 e 1889 devem ser CONSOLIDADOS.

## Estrutura do Informe (Abas)

| Aba | Conteudo | Fonte | Codigo |
|-----|----------|-------|--------|
| 1 | Rendimentos tributaveis, IRRF | IRT_CEDULA_D | 3208 |
| 2-5 | Deducoes, previdencia, pensao, outros | IRT_CEDULA_D | - |
| 6 | RRA (Rendimentos Recebidos Acumuladamente) | SAFX181 | 1889 |

## Opcoes de Solucao (Nao Implementadas)

### Opcao 1 - Java (Recomendada)

Modificar `CedulaDDAO.obterInformes()` para tratar 3208 e 1889 como mesmo grupo:

```java
// Normalizar grupo para codigos que devem ser consolidados
Set<String> CONSOLIDATE_CODES = Set.of("3208", "1889");
String grupoKey = CONSOLIDATE_CODES.contains(codRetencao)
    ? "RENTAL_CONSOLIDATED" : campos[7];
```

**Pros**: Minimal risk, nao quebra 3208 vs 5204, reversivel
**Cons**: Requer manter lista de codigos consolidaveis

### Opcao 2 - PL/SQL

Modificar `pkg_comprovante_retencoes.pck` cursor com CASE WHEN para normalizar grupo_darf.

**Pros**: Single-source-of-truth
**Cons**: Requer editar 3 UNION clauses, ISO-8859-1

### Opcao 3 - Jasper (Nao Recomendada)

Report nao deve conter regras de negocio.

## Licoes Aprendidas

1. **Consolidacao vs Separacao**: Codigos de retencao podem precisar ser consolidados (3208+1889)
   OU separados (3208+5204) dependendo da regra fiscal. A chave de agrupamento deve considerar
   relacoes entre codigos, nao apenas grupo_retencao isolado.

2. **Fluxo PL/SQL→CSV→Java→Jasper**: Mudanca em qualquer camada afeta o resultado. A camada
   Java (CedulaDDAO) e o ponto ideal para consolidacao pois permite logica condicional sem
   alterar a estrutura de dados.

3. **SAFX181 e aba 6**: A aba 6 (RRA) do informe so e preenchida por importacao SAFX181,
   que depende da SAFX15 (cadastro funcionario/terceiro) e campo 46 (vinculo empregaticio).

4. **Informe de Rendimentos tem multiplas abas**: Cada aba tem fonte de dados diferente.
   Consolidar informes requer garantir que todas as abas sejam preenchidas corretamente
   no documento unico.
