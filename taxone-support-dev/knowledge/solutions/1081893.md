# ORA-06502 na Consolidacao de Documentos UTILITIES (Mapa Resumo ICMS)

## Root Cause

O package `SAF_ICMS_MAPA_RES_FPROC` processa a consolidacao de documentos de telecomunicacoes
(SAFX42/SAFX43) para o Mapa Resumo de ICMS. O processo agrupa e soma valores de multiplos
documentos fiscais em uma unica nota consolidada.

O record type `trTotValores` (linha ~855) declarava todos os campos como `NUMBER(17,2)`,
limitando valores intermediarios a max 999.999.999.999.999,99. Para clientes de telecom com
alto volume (como HUGHES com 364+ documentos/mes na Filial 400/RJ), o acumulo de valores
durante a consolidacao ultrapassava essa precisao.

O erro se manifestava no `FORALL INSERT` em `x07_trib_docfis` (linha ~3345) e propagava pelo
call stack: `gravaNota` → `setItMerc`/`setNfCapaTrib` → ponto de acumulacao.

**Comportamento diagnostico:** O erro "migrava" entre documentos — ao excluir a NF que falhava
(18466-2), o erro aparecia na proxima NF (18465-2). Isso confirma overflow no acumulo total,
nao em um documento especifico. Em homologacao (menor volume), o processo completava sem erro.

**Stack trace producao:**
```
ORA-06502: PL/SQL: numeric or value error: number precision too large
ORA-06512: at USER_TAXONE_HUGHES.SAF_ICMS_MAPA_RES_FPROC, line 2555
ORA-06512: at USER_TAXONE_HUGHES.SAF_ICMS_MAPA_RES_FPROC, line 2815
ORA-06512: at USER_TAXONE_HUGHES.SAF_ICMS_MAPA_RES_FPROC, line 3347
```

## Solucao

4 alteracoes no arquivo `SAF_ICMS_MAPA_RES_FPROC.pck`:

### 1. trTotValores: NUMBER(17,2) → NUMBER (linha ~855)

Remocao da restricao de precisao no record type usado para calculos intermediarios.
NUMBER sem precisao aceita ate 38 digitos, eliminando overflow em acumulacoes.

```sql
-- ANTES
TYPE trTotValores IS RECORD (vlrservico   NUMBER(17,2),
                              vlrdesconto  NUMBER(17,2),
                              ...);

-- DEPOIS
TYPE trTotValores IS RECORD (vlrservico   NUMBER,
                              vlrdesconto  NUMBER,
                              ...);
```

### 2. ROUND nos pontos de acumulacao de tributos (linha ~2915)

Adicionado ROUND na procedure `setNfCapaTrib` para garantir que valores acumulados
respeitem a precisao das colunas destino antes do INSERT:

```sql
-- ANTES
aDoctTrib(vIdx).vlr_tributo  := nvl(aDoctTrib(vIdx).vlr_tributo, 0) + nvl(pVlr_tributo, 0);
aDoctTrib(vIdx).aliq_tributo := greatest(nvl(pAliq_tributo, 0), nvl(aDoctTrib(vIdx).aliq_tributo, 0));

-- DEPOIS
aDoctTrib(vIdx).vlr_tributo  := round(nvl(aDoctTrib(vIdx).vlr_tributo, 0) + nvl(pVlr_tributo, 0), 2);
aDoctTrib(vIdx).aliq_tributo := round(greatest(nvl(pAliq_tributo, 0), nvl(aDoctTrib(vIdx).aliq_tributo, 0)), 4);
```

### 3. ROUND nos pontos de acumulacao de bases (3 procedures)

Adicionado `ROUND(..., 2)` em `setNfCapaBase`, `setItMercBase` e `setItServBase`:

```sql
-- ANTES (em cada procedure)
aXxxBase(vIdx).vlr_base := nvl(aXxxBase(vIdx).vlr_base, 0) + nvl(pVlr_base, 0);

-- DEPOIS
aXxxBase(vIdx).vlr_base := round(nvl(aXxxBase(vIdx).vlr_base, 0) + nvl(pVlr_base, 0), 2);
```

### 4. Exception handler diagnostico no FORALL INSERT (linha ~3345)

Adicionado bloco BEGIN...EXCEPTION...END no FORALL INSERT de `x07_trib_docfis`
com logging individual de cada elemento do array antes do re-raise:

```sql
begin
   forall i in aDoctTrib.first..aDoctTrib.last
      insert /*+ append */ into x07_trib_docfis values aDoctTrib(i);
exception
   when others then
      for j in aDoctTrib.first..aDoctTrib.last loop
         if aDoctTrib.exists(j) then
            lib_proc.add_log('Tributo idx=' || j || ' vlr=' || aDoctTrib(j).vlr_tributo
               || ' aliq=' || aDoctTrib(j).aliq_tributo || ' cod=' || aDoctTrib(j).cod_tributo, 0);
         end if;
      end loop;
      raise;
end;
```

## Licoes Aprendidas

### Overflow em acumulacao PL/SQL com %ROWTYPE

Variaveis declaradas como `table of TABELA%rowtype` herdam a precisao das colunas.
Quando usadas para **acumular** valores de multiplos registros (SUM iterativo), podem
estourar NUMBER(17,2) em cenarios de alto volume. Solucao: usar NUMBER sem precisao
para variaveis intermediarias, e ROUND antes do INSERT.

**Regra:** Em packages de consolidacao/totalizacao, NUNCA usar NUMBER(p,s) em records
intermediarios. Usar NUMBER puro e aplicar ROUND somente no ponto de gravacao.

### Erro que "migra" entre documentos = overflow no acumulo

Quando o ORA-06502 aparece em um documento diferente apos excluir o anterior, o problema
nao esta no documento especifico, mas no **total acumulado**. O documento que falha e
simplesmente aquele que faz o acumulo ultrapassar o limite. Teste de confirmacao:
executar em ambiente com menor volume (homologacao) — se funciona, e overflow.

### Exception handler em FORALL para diagnostico

O FORALL INSERT nao identifica qual elemento do array causou o erro. Adicionar
BEGIN...EXCEPTION com loop de logging permite identificar os valores individuais
sem alterar o comportamento (re-raise preserva o erro original).

### Observacao DBA: SUM de aliquotas no cursor

O cursor `cItemTel` faz `SUM(x43.vlr_aliq_icms_fecep)` — somando aliquotas que
deveriam usar MAX (aliquota e percentual, nao valor). Isso pode gerar overflow
secundario em NUMBER(7,4) (max 999,9999). Nesta correcao, o ROUND(4) protege,
mas a correcao ideal seria trocar SUM por MAX no cursor para campos de aliquota.
Deixado como melhoria futura por ser alteracao de maior risco.
