# Comprovante Anual Retencao - Email nao enviado e PDF em branco

## Root Cause

Tres problemas distintos na geracao e envio do Comprovante Anual/Mensal de Retencao:

1. **Email retornava sucesso quando PDF nao existia**: O metodo `enviarEMail()` nao
   verificava se o arquivo PDF foi realmente gerado antes de tentar envia-lo. Retornava
   sucesso mesmo sem arquivo.

2. **PDF exportado com impressao = null**: O metodo `gravarRel()` em `GerarRelatorio.java`
   exportava o PDF passando `impressao = null`, resultando em arquivo em branco.

3. **Falta de verificacao do retorno de gravarRel()**: O codigo chamava `gravarRel()` e
   prosseguia para `enviarEMail()` sem checar se a geracao do PDF teve sucesso.

## Solucao

- **GeraComprovanteAnualRetencao.java**: Adicionada verificacao do retorno de `gravarRel()`
  antes de chamar `enviarEMail()`. Se PDF nao for gerado, loga erro e nao tenta enviar.

- **GeraComprovanteMensalRetencao.java**: Mesma correcao aplicada.

- **GerarRelatorio.java**: Corrigido parametro `impressao` para valor correto ao exportar PDF.

## Arquivos Modificados

1. `artifacts/java/.../business/GeraComprovanteAnualRetencao.java` (+15/-12)
2. `artifacts/java/.../business/GeraComprovanteMensalRetencao.java` (+14/-11)
3. `artifacts/java/.../util/GerarRelatorio.java` (+9/-1)

## Licoes Aprendidas

1. **Sempre verificar existencia do arquivo antes de enviar**: Em fluxos de gerar+enviar,
   a geracao pode falhar silenciosamente. Checar retorno E existencia do arquivo.

2. **Grandes volumes + email**: O processamento por CNPJ para grandes volumes (como Equinor)
   pode exacerbar problemas que nao aparecem em volumes menores. Testar com multiplos
   beneficiarios.

3. **Modulo Java (Edit tool OK)**: Diferente dos PL/SQL (ISO-8859-1), os arquivos Java
   sao UTF-8 e podem ser editados com a Edit tool normalmente.
