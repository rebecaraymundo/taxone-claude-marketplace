# DART/TUPPERWARE - Alteracao de dados NF via tela Java causa erro silencioso

## Root Cause

Na tela Java de edicao de documento fiscal (taxone_modules_t1dw), ao alterar dados
de uma NF existente, a procedure `PKG_TMP_DOCTO_FISCAL_FPROC.GravarX07` chamava
`SAF_OL_DW_X07` com parametro `'I'` (Insert) em vez de `'A'` (Alteracao).

Para documentos ja existentes nas tabelas definitivas (cenario PROD), isso causava
`DUP_VAL_ON_INDEX`. Alem disso, o Java (`TempTableService`) nao verificava o
status de retorno da procedure, retornando sempre HTTP 200 ao frontend, que
exibia "No momento nenhum erro na nota fiscal" mesmo com falha na gravacao.

## Por que funciona em suporte mas nao em PROD

Na base de suporte, o documento pode nao existir nas tabelas definitivas (INSERT
funciona). Em PROD, documentos antigos ja existem, causando violacao de chave unica.

## Solucao

### PL/SQL (taxone_dw - PR #12194)

1. **GravarX07 (linha 6979)**: Alterado `'I'` para `'A'` na chamada SAF_OL_DW_X07
   - `'A'` seta `P_IND_SP_REGISTRO_W='S'` (existente) em d_ix07.sql (linha 422-428)
   - `'I'` setava `'U'` (unknown/new), causando INSERT em registro existente

2. **GravarNotaFiscal**: Adicionado `ROLLBACK` no `WHEN OTHERS` para consistencia

3. **GravarNotaFiscal**: Adicionado log de sucesso (cod_erro=60906) em `log_det_proc`

### Java (taxone_modules_t1dw - PR #161)

4. **TempTableService.convertFromTemporaryToFinal**: Adicionada verificacao de
   `result.getStatus() < 0` apos chamada da procedure. Lanca `InvoiceSaveException`
   com `numProcess` para diagnostico.

5. **InvoiceSaveException**: Nova excecao `@ResponseStatus(INTERNAL_SERVER_ERROR)`

## Duas Arquiteturas Paralelas

O TAX ONE tem 2 caminhos para gravar documentos fiscais:
- **PowerBuilder**: Usa `GravaNotaDW.pck` (procedure dinamica via `user_arguments`)
- **Java/Spring**: Usa `PKG_TMP_DOCTO_FISCAL_FPROC` (chamada direta via `SimpleJdbcCall`)

Este bug afeta APENAS o caminho Java. O `GravaNotaDW.pck` tem logica separada.

## Licoes Aprendidas

1. **Identificar a arquitetura correta**: Sempre confirmar se o cliente usa tela
   PowerBuilder ou Java antes de analisar. A tela Java usa package diferente.

2. **PKG_TMP_DOCTO_FISCAL_FPROC vs GravaNotaDW**: Sao caminhos independentes.
   Correcoes em um nao afetam o outro.

3. **SimpleJdbcCall nao lanca excecao**: O DAO retorna ConversionResultDTO com
   status, mas o service nao verificava. Sempre checar retorno de procedures.

4. **SAF_OL_DW_X07 parametro 'I' vs 'A'**: Linha 422-428 de d_ix07.sql mostra
   que 'A' marca registro como existente (UPDATE), 'I' como novo (INSERT).
