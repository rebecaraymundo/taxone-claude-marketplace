# [Interfaces] [SAP – Namespace/OBTI] [Web Service] Erro de Comunicação no envio

### Problema

Falha no envio de arquivos do OBTI para o OBI; mensagem "Client HTTP - erro de comunicação" após configuração de criptografia e webservice no ambiente OBTI.

### Causa Raiz

Ambiente está com bloqueio de saída pelo proxy/firewall, evidenciado pelo retorno _**HTTP/1.1 403 Forbidden com cabeçalho X-Squid-Error: ERR_ACCESS_DENIED**_. Esse status indica negação pelo servidor proxy Squid devido a regra de segurança, como por exemplo: uso de VPN, cache/proxy desatualizado, política de acesso, ou ausência de liberação do destino.

### Resolução

  1. Verificações rápidas


  * Limpar cache e cookies do navegador (se houver etapa via browser) e executar ipconfig /flushdns no Windows.
  * Desativar temporariamente VPN/agent ZTNA para teste controlado.
  * Efetuar teste sem proxy: retire a configuração de proxy na tela de webservices do OBTI e tente o envio novamente. Se funcionar sem proxy, o bloqueio está na cadeia proxy/firewall.


  2. Liberações de rede/proxy


  * Solicitar à infraestrutura a liberação/allowlist do host nas saídas HTTP/HTTPS, conforme o endpoint utilizado, e das portas correspondentes.
  * Garantir que o proxy permita o método HTTP necessário (GET/POST) e o caminho /gateway/healthcheck/status e os demais endpoints do OBI.
  * Revisar regras de autenticação do proxy para o usuário/serviço do OBTI (dispensa de autenticação ou credenciais corretas).


  3. Testes técnicos de conectividade


  * SAP SM59 (tipo G – HTTP Connection to External Server):
    * Target Host: uatxx.obi.thomsonreuters.com
    * Path Prefix: /gateway/healthcheck/status
    * Proxy: preencher somente se a saída obrigar proxy; caso contrário, deixar em branco.
    * Executar Connection Test e verificar retorno HTTP 200.


  4. Higienização de cache/proxy


  * Solicitar limpeza/refresh de cache do Squid e validação de regras atuais.
  * Sincronizar regras entre ambientes (UAT/PRD) para evitar heranças de políticas antigas.


  5. Testar novamente o processo de envio


  * Após liberação e validações, reenviar os arquivos do OBTI para o OBI e confirmar ausência do erro


#### Próximos Passos

  * Alinhar com a equipe de infraestrutura a criação de regra explícita no proxy/firewall permitindo o tráfego do OBTI para o OBI nas portas necessárias.
  * Registrar e analisar logs: firewall e trace HTTP do OBTI durante um envio de teste.
  * Padronizar a configuração SM59 no OBTI (com e sem proxy) e documentar qual rota é oficial para o ambiente OBI.
  * Validar periodicamente o endpoint /gateway/healthcheck/status para rápida detecção de indisponibilidades.


#### Dicas e Recomendações

  * Manter uma allowlist oficial de domínios/URLs do OBI utilizada pelo OBTI, com revisão a cada mudança de ambiente.
  * Evitar uso de VPN durante janelas de integração, salvo se as rotas e exceções estiverem corretamente configuradas no proxy.
  * Documentar as credenciais/política de autenticação do proxy para o usuário de serviço do OBTI e revisar expiração periódica.
  * Monitorar com checks automatizados o healthcheck do OBI e alertar a equipe em caso de retorno diferente de 200.
  * Sempre que possível, testar conectividade diretamente do servidor do OBTI (sem hops locais) para isolar problemas de estação.
